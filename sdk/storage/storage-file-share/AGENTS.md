# storage-file-share Agent Guidance

This file provides package-specific guidance for AI agents working in `sdk/storage/storage-file-share/`.

## Package Overview

`@azure/storage-file-share` is the Azure Files client library. It exposes:

| Client | Purpose |
| --- | --- |
| `ShareServiceClient` | Account-level operations (list shares, get/set service properties) |
| `ShareClient` | Share-level operations (create, delete, snapshots, permissions, quotas) |
| `ShareDirectoryClient` | Directory operations (create, delete, list files and subdirectories) |
| `ShareFileClient` | File operations (create, upload, download, delete, copy, lease) |
| `ShareLeaseClient` | Lease acquisition and management on shares and files |

## Rules for AI Agents

### Rule 1: Do Not Edit Generated Code

Do not manually edit any files under `src/generated/`. These are regenerated by AutoRest and any manual change will be overwritten. To change generated behaviour, update the AutoRest configuration and regenerate.

### Rule 2: Preserve Directory Hierarchy Semantics

Azure Files is a hierarchical service. Directory existence and parent-child relationships must be explicit. Do not silently create missing parent directories unless the caller opts in (e.g., a `createIfNotExists`-style helper). Path normalization must be consistent with existing helpers in `utils/`.

### Rule 3: Respect SMB-Compatible File Attributes

Azure Files supports Windows SMB file system attributes (`FileSystemAttributes`). When adding or modifying operations that touch file properties, preserve and thread through `fileAttributes`, `fileCreatedOn`, `fileLastWriteOn`, and related fields. Do not drop them silently.

### Rule 4: Browser and Node.js Parity

`FileDownloadResponse` has browser and react-native variants (`*-browser.mts`, `*-react-native.mts`). When changing download response handling, update all variants.

### Rule 5: Reuse `storage-common` Helpers

Shared utilities (pipeline builders, `StorageSharedKeyCredential`, `AnonymousCredential`, retry policy) live in `@azure/storage-common`. Do not re-implement them here.

## Key Concepts

- **Share hierarchy:** Account → Share → Directory (nested) → File. Directories must exist before files can be created inside them.
- **Snapshots:** Shares support point-in-time read-only snapshots; snapshot timestamp is part of the URL. `ShareClient.createSnapshot()` returns the snapshot identifier.
- **SMB attributes:** Files carry Windows-compatible attributes (`Hidden`, `ReadOnly`, `Archive`, etc.) managed via `FileSystemAttributes`.
- **SAS tokens:** Generated client-side using `generateFileSASQueryParameters` or `generateAccountSASQueryParameters`; sign with `StorageSharedKeyCredential`.
- **Leases:** Managed via `ShareLeaseClient`; infinite-duration leases are available for files (unlike blobs). A lease ID must be included in mutating operations on a leased resource.
- **Copy and upload:** Large file uploads use `uploadStream` / `uploadFile` helpers that split data into range-upload calls. Copy operations are async and tracked via a copy ID.
