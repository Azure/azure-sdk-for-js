# storage-blob Agent Guidance

This file provides package-specific guidance for AI agents working in `sdk/storage/storage-blob/`.

## Package Overview

`@azure/storage-blob` is the Azure Blob Storage client library. It exposes:

| Client | Purpose |
| --- | --- |
| `BlobServiceClient` | Account-level operations (list containers, get properties, generate SAS) |
| `ContainerClient` | Container-level operations (create, delete, list blobs) |
| `BlobClient` | Base blob operations (download, delete, get properties, copy, lease) |
| `BlockBlobClient` | Block blob upload and commit operations |
| `AppendBlobClient` | Append blob create and append operations |
| `PageBlobClient` | Page blob create and page write/resize operations |
| `BlobBatchClient` | Batched blob delete and set-tier requests |
| `BlobLeaseClient` | Lease acquisition and management on blobs and containers |

## Rules for AI Agents

### Rule 1: Do Not Edit Generated Code

Do not manually edit any files under `src/generated/`. These are regenerated by AutoRest and any manual change will be overwritten. To change generated behaviour, update the AutoRest configuration or models in `swagger/` and regenerate.

### Rule 2: Preserve Block Blob Two-Phase Commit Semantics

Block blobs become visible only after `commitBlockList`. High-level upload helpers (e.g., `uploadData`, `uploadStream`, `uploadFile`) stage blocks and then commit. Do not short-circuit or collapse these two phases, and do not expose uncommitted block data to callers.

### Rule 3: Preserve Append Blob Order Safety

Append blobs accumulate blocks sequentially. Use `appendPositionConditions` / ETag conditions in `appendBlock` when concurrent-write safety is required. Do not remove existing position guards in the client.

### Rule 4: Browser and Node.js Parity

Each response type that reads a stream has a browser variant (`*-browser.mts`) and a Node.js variant (`*.ts`). When changing response handling, update both variants and the react-native variant where one exists.

### Rule 5: Reuse `storage-common` Helpers

Shared utilities (pipeline builders, `StorageSharedKeyCredential`, `AnonymousCredential`, retry policy, SAS helpers, `BufferScheduler`) live in `@azure/storage-common`. Do not re-implement them here.

## Key Concepts

- **Block commit flow:** `stageBlock` → `commitBlockList`. The committed blob content is invisible until `commitBlockList` succeeds.
- **SAS tokens:** Generated client-side using `generateBlobSASQueryParameters` / `generateAccountSASQueryParameters`; sign with `StorageSharedKeyCredential` or a user-delegation key.
- **Leases:** Managed via `BlobLeaseClient`; a lease ID must be included in mutating operations on a leased blob or container.
- **Tags and metadata:** Tags are key-value string pairs indexed for search; metadata is arbitrary string key-value pairs attached to a resource. They have separate get/set operations.
- **Soft delete:** Deleted blobs can be recovered within the retention window using `BlobClient.undelete()`.
