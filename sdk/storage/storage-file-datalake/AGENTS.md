# storage-file-datalake Agent Guidance

This file provides package-specific guidance for AI agents working in `sdk/storage/storage-file-datalake/`.

## Package Overview

`@azure/storage-file-datalake` is the Azure Data Lake Storage Gen2 (ADLS Gen2) client library. It is built on top of `@azure/storage-blob` and exposes a hierarchical namespace (HNS) API. It exposes:

| Client | Purpose |
| --- | --- |
| `DataLakeServiceClient` | Account-level operations (list file systems, get/set service properties) |
| `DataLakeFileSystemClient` | File system (container) operations (create, delete, list paths) |
| `DataLakeDirectoryClient` | Directory operations (create, delete, rename, ACL management) |
| `DataLakeFileClient` | File operations (create, append, flush, read, delete, rename, query) |
| `DataLakeLeaseClient` | Lease acquisition and management on file systems, directories, and files |

## Rules for AI Agents

### Rule 1: Do Not Edit Generated Code

Do not manually edit any files under `src/generated/`. These are regenerated by AutoRest and any manual change will be overwritten. To change generated behaviour, update the AutoRest configuration and regenerate.

### Rule 2: ADLS Gen2 Is Built on Blob Storage â€” Respect the Layering

`@azure/storage-file-datalake` re-uses `@azure/storage-blob` clients internally (e.g., `BlobClient`, `BlockBlobClient`). Operations that don't have a native ADLS Gen2 REST endpoint delegate to the blob layer. Changes to blob-layer semantics can break ADLS Gen2 behaviour. When modifying blob interaction in this package, verify that the ADLS-level contract is preserved.

### Rule 3: Preserve ACL and POSIX Permission Semantics

ADLS Gen2 supports POSIX-style access control lists (ACLs) with `owner`, `group`, `permissions` (`rwxrwxrwx`), and named ACL entries. These are separate from Azure RBAC and blob metadata. Do not conflate or drop ACL fields when working on path create/update operations.

### Rule 4: Recursive ACL Changes Use `PathChangeAccessControlRecursiveOptions`

Recursive ACL changes (`setAccessControlRecursive`, `updateAccessControlRecursive`, `removeAccessControlRecursive`) are long-running operations with continuation tokens. They use `PathChangeAccessControlRecursiveOptions` and return `PathChangeAccessControlRecursiveResponse` with batch counters. Do not simplify these to single-shot operations.

### Rule 5: Model Transformations Live in `transforms.ts`

Blob model responses are translated to ADLS Gen2 public models in `transforms.ts`. When adding new properties to public models, add the mapping here rather than in individual client methods.

### Rule 6: Reuse `storage-common` Helpers

Shared utilities (pipeline builders, `AnonymousCredential`, retry policy, `BufferScheduler`) live in `@azure/storage-common`. Do not re-implement them here.

## Key Concepts

- **Hierarchical namespace (HNS):** Directories are first-class objects with atomic rename and move. File system is the equivalent of a blob container with HNS enabled.
- **Append and flush:** ADLS Gen2 file writes are two-phase: `append` stages data at an offset, then `flush` commits all staged data up to a position. Do not expose partially flushed data as complete.
- **ACLs:** Each path has an owner, owning group, POSIX permission bits, and optional named ACL entries for users and groups. ACLs are separate from metadata.
- **Continuation tokens:** Recursive ACL operations and list operations return continuation tokens for paging. Preserve and thread through these tokens correctly.
- **SAS tokens:** Generated client-side using `generateDataLakeSASQueryParameters` or `generateAccountSASQueryParameters`; sign with `StorageSharedKeyCredential`.
- **Leases:** Managed via `DataLakeLeaseClient`; a lease ID must be included in mutating operations on a leased resource.
